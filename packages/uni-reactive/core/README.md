# Uni Reactive Core

مكتبة `uni-reactive` هي طبقة توحيد (Abstraction Layer) لطريقة التعامل مع الحالة التفاعلية عبر أطر مختلفة، بدون استبدال النظام التفاعلي الأصلي لأي إطار.

## الفكرة العامة

الهدف من المكتبة هو إعطاء **عقد موحّد** تتعامل معه المكتبات المشتركة (مثل Local DB أو Cache أو Domain Services)، بحيث لا تحتاج لمعرفة إن كان التطبيق يعمل على Vue أو Solid أو Angular Signals أو أي نظام Reactive آخر.

بدلًا من ذلك:
- المكتبات تتعامل مع واجهة واحدة واضحة.
- التطبيق هو من يحدد Plugin الإطار عند التشغيل.
- القيمة التفاعلية تُعاد للمستهلك **بنفس النوع الأصلي Native** للإطار.

> النتيجة: منطق الأعمال يبقى مستقلًا عن الإطار، وتجربة المطور النهائي تبقى طبيعية داخل إطاره.

## الفلسفة المعمارية

### 1) Core محايد بالكامل
طبقة `Core` لا تعرف شيئًا عن Vue أو React أو غيرها. مسؤوليتها:
- تعريف العقود (Contracts) فقط.
- تنظيم دورة حياة الربط مع Plugin.
- ضمان أن العمليات المسموحة موحّدة (create / update / read contractually).

### 2) Plugin لكل إطار
كل إطار يقدّم Plugin خاصًا به كـ Adapter:
- يترجم عقود Core إلى primitives أصلية للإطار.
- يضمن أن الإنشاء والتحديث يتمان بأسلوب الإطار نفسه.
- يعيد النوع التفاعلي الحقيقي بدون Wrapper إضافي.

### 3) Binding مركزي على مستوى التطبيق
الربط يتم مرة واحدة عند بدء التطبيق:
- التطبيق يختار Plugin المناسب.
- يتم تسجيله في Core كـ Active Runtime.
- كل المكتبات اللاحقة تستخدم نفس الربط دون أي اختيار محلي.

هذا يمنع تضارب البيئات ويجعل القرار المعماري واضحًا ومركزيًا.

## أدوار الطبقات في النظام

### طبقة Core
- تعريف الواجهة العامة للمكتبات المشتركة.
- فرض Type Contracts مستقرة.
- إدارة حالة "Plugin محمّل/غير محمّل".
- توفير نقطة تكامل واحدة لأي Plugin جديد مستقبلًا.

### طبقة Plugins
- تنفيذ العقود باستخدام Reactive API الأصلية للإطار.
- احترام سلوك التحديث والاشتقاق (derivation) وفق الإطار.
- عزل تفاصيل الإطار داخل حدود Plugin فقط.

### طبقة التطبيق (Application Composition)
- اختيار Plugin المناسب حسب الإطار المستخدم.
- تحميله مرة واحدة.
- تمرير الـ runtime الجاهز إلى أجزاء التطبيق والمكتبات.

### طبقة المكتبات المشتركة (Shared Libraries)
- استخدام عقود Core فقط.
- إنشاء الحالة وتعديلها عبر واجهة موحدة.
- عدم استيراد أي كود من إطار محدد.

## كيف تتفاعل المكتبات مع النظام دون معرفة الإطار

التدفق المفاهيمي:
1. التطبيق يفعّل Plugin الإطار عند الإقلاع.
2. مكتبة مشتركة تطلب إنشاء حالة عبر Core Contract.
3. Core يوجّه الطلب إلى Plugin الفعّال.
4. Plugin ينشئ reactive primitive الأصلية.
5. المكتبة تعيد الناتج للمستهلك النهائي كما هو Native.
6. التعديلات اللاحقة تمر عبر نفس العقد الموحد وتبقى متوافقة مع الإطار.

بهذا النموذج، نفس كود المكتبة المشتركة يعمل عبر أطر متعددة بدون fork أو if-else مرتبط بالإطار.

## ضمانات الجودة

- **قابلية الاختبار:**
  - اختبار Core بمعزل عبر Mock Plugins.
  - اختبار كل Plugin على حدة بعقود توافق واضحة.
- **سهولة الفهم:**
  - Core صغير ومسؤولياته محددة.
  - Plugins واضحة كجسور تكامل.
- **Type Safety:**
  - العقود typed generically.
  - نوع القيمة المرجعة مرتبط بنوع Plugin المستخدم.

## اتجاهات التوسع المستقبلية

- إضافة Plugins جديدة (مثل Signals-based runtimes أو Stores متقدمة) دون تعديل منطق المكتبات الحالية.
- دعم Capabilities اختيارية داخل Plugin (مثل batching أو transactions) عبر آلية feature negotiation.
- إدخال طبقة تشخيص اختيارية (Dev Diagnostics) للتحقق من صحة الربط أثناء التطوير.
- توفير حزم Plugin منفصلة لكل إطار للحفاظ على حجم اعتماد منخفض.
- دعم إصدارات متعددة من نفس الإطار عبر توافق عقدي واضح.

## حدود واعية للتصميم

هذه المكتبة **لا**:
- توحّد semantics الداخلية لجميع الأنظمة التفاعلية.
- تفرض نموذج reactive جديد على المطور.
- تستبدل APIs الأصلية للإطار.

هي فقط توحّد **واجهة التكامل** للمكتبات المشتركة، وتترك التنفيذ والسلوك النهائي Native داخل كل إطار.
